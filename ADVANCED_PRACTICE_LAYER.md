# שכבת תרגול מתקדמת — גרסה 1.1

## 1. מטרת השכבה

שכבת התרגול נועדה להפוך ידע מהקורס ליכולת מקצועית מוכחת: לתכנן שינוי, לבנות אותו בעזרת כלי פיתוח AI, לבדוק את התוצאה, לתעד אותה ולהציג אותה כתוצר שאפשר להסביר ללקוח.

השכבה מיועדת ללומד שסיים את החלקים הרלוונטיים בקורס ורוצה לתרגל עבודה ברמה גבוהה יותר. היא כוללת **שלושה תרגולים מרכזיים בלבד**.

---

## 2. למה השכבה לא משנה את הקורס

- 36 השיעורים, הסדר שלהם והפרויקט המרכזי נשארים ללא שינוי.
- התרגולים משתמשים רק בידע שכבר נלמד בקורס.
- השכבה היא חומר נלווה לתרגול, לא חלק חדש ולא מסלול חובה נוסף.
- אין צורך להוסיף שיעורים, לשנות את האתר או לבחור Framework חדש.
- כל תרגול מסתיים בתוצר מוגדר ובנקודת עצירה ברורה.

---

## 3. שלושה עקרונות עבודה מתקדמים

### 1. שינוי קטן ומוגדר

מתחילים בבעיה ברורה, מגדירים מה ייחשב הצלחה ומבצעים שינוי אחד בלבד. לא מרחיבים את המשימה תוך כדי עבודה.

### 2. בדיקה לפני קבלה

פלט של Codex או Claude Code הוא הצעה שצריך לבדוק. קוראים את השינוי, בודקים את ה־diff, מריצים בדיקות ומוודאים שלא נפגע דבר קיים לפני commit.

### 3. תיעוד שמוכיח עבודה

כל תרגול משאיר ראיות קצרות: Plan, מה השתנה, מה נבדק, איזו בעיה נמצאה ומה התוצאה. התיעוד צריך לאפשר לאדם אחר להבין את העבודה בלי לנחש.

---

## 4. שלושת התרגולים המרכזיים

### תרגול 1: עבודה מקצועית עם Codex, Claude Code ו־GitHub

**מטרה:** לתרגל Workflow מבוקר של שינוי תוכנה קטן.

**המשימה:** לבחור תיקון או שיפור קטן בפרויקט תרגול שאינו דורש שינוי ארכיטקטורה.

**מהלך העבודה:**

1. לכתוב Plan קצר: מטרה, הקשר, מגבלות ו־Done When.
2. לבדוק את מצב Git וליצור נקודת עבודה בטוחה בהתאם לפרויקט.
3. להשתמש ב־Codex או ב־Claude Code לביצוע השינוי הקטן.
4. להשתמש בכלי השני או בבדיקה אנושית לסקירת השינוי וה־diff.
5. להריץ את הבדיקות המתאימות ולבדוק את ההתנהגות בפועל.
6. לתקן רק בעיות שנמצאו ושייכות למשימה.
7. ליצור commit ממוקד עם הודעה ברורה.

**תוצר:** Plan, שינוי עובד, תוצאות בדיקה, סיכום Review ו־commit אחד ממוקד.

**התרגול הושלם כאשר:** השינוי עומד ב־Done When, הבדיקות עוברות, ה־diff נבדק ואין קבצים לא קשורים ב־commit.

---

### תרגול 2: חבילת תוצר לקוח

**מטרה:** לתרגל מעבר מפתרון טכני להסבר עסקי ומסירה מקצועית.

**המשימה:** לבחור פתרון אחד מהפרויקט המרכזי ולהכין עבורו חבילת מסירה קצרה.

**החבילה תכלול:**

1. **מסמך אפיון קצר:** הבעיה, המשתמש, הקלט, הפלט, גבולות הפתרון ומדד הצלחה אחד או שניים.
2. **דמו או הסבר פתרון:** מה המערכת עושה, איך התהליך עובד ומה האדם צריך לבדוק.
3. **הצעת מחיר בסיסית:** היקף, תוצרים, מה לא כלול, זמן משוער ומודל שירות בסיסי.
4. **תוכנית מסירה ותחזוקה:** גישה, הדרכה, בדיקות, אחריות, דיווח תקלות ושיפורים עתידיים.

**תוצר:** חבילת לקוח קצרה שאפשר להציג בתוך 5–10 דקות.

**התרגול הושלם כאשר:** אדם שאינו מכיר את הפרויקט יכול להבין מה הבעיה, מה הפתרון, מה יקבל הלקוח ומהם הגבולות.

---

### תרגול 3: בדיקות ואיכות

**מטרה:** לתרגל מחזור מקצועי של בדיקה, איתור בעיה, תיקון ותיעוד.

**המשימה:** לבחור זרימה מרכזית אחת בפתרון ולבדוק אותה מתחילתה ועד סופה.

**מהלך העבודה:**

1. להגדיר מה אמור לקרות ומה ייחשב כישלון.
2. לבצע בדיקה רגילה ובדיקת קצה אחת.
3. לזהות בעיה אמיתית או נקודת חולשה שניתן לשחזר.
4. לתעד את המצב לפני התיקון.
5. לבצע תיקון קטן וממוקד.
6. להריץ שוב את הבדיקה המקורית ובדיקת Regression רלוונטית.
7. לתעד מה נבדק, מה השתנה ומה עדיין לא מכוסה.

**תוצר:** דוח בדיקה קצר עם תרחיש, תוצאה צפויה, תוצאה בפועל, תיקון ותוצאת בדיקה חוזרת.

**התרגול הושלם כאשר:** הבעיה ניתנת להסבר ולשחזור, התיקון נבדק ולא נמצאה פגיעה ידועה בהתנהגות הקיימת.

## 5. Rubric קצר להגדרת רמה טובה

לכל קריטריון נותנים ציון 0–2:

| קריטריון | 0 — חסר | 1 — חלקי | 2 — רמה טובה |
|---|---|---|---|
| תכנון והיקף | אין מטרה או גבולות | המטרה קיימת אך אינה מדויקת | מטרה, מגבלות ו־Done When ברורים |
| ביצוע | התוצר אינו עובד | עובד חלקית או דורש הסבר רב | עובד בהתאם להגדרה ובשינוי ממוקד |
| בדיקות | לא בוצעו בדיקות | בוצעה בדיקה בסיסית בלבד | קיימות בדיקה רגילה, בדיקת קצה ותוצאה מתועדת |
| Review ותיעוד | לא נבדק ה־diff ואין תיעוד | התיעוד חלקי | השינוי נסקר, ההחלטות והבדיקות מוסברות |
| ערך מקצועי | לא ברור למי התוצר עוזר | הערך מובן חלקית | אפשר להסביר את הערך, הגבולות והמסירה ללקוח |

**רמה טובה:** לפחות 8 מתוך 10, ללא ציון 0 בתכנון, בבדיקות או בתיעוד.

## 6. מה לא נכנס כרגע

- שיעורים חדשים או שינוי מבנה הקורס.
- LangChain, CrewAI או Framework חובה אחר.
- Multi-Agent מתקדם, תשתיות מורכבות או אוטומציה רחבה.
- מערכת ציונים, תעודות או בדיקה אוטומטית.
- התחברות, מסד נתונים או פיצ'ר חדש באתר.
- CI/CD מתקדם, אבטחת Production מלאה או התחייבות רגולטורית.
- יותר משלושת התרגולים המוגדרים במסמך זה.

---

## 7. איך לא להפוך את השכבה לקורס אינסופי

- מבצעים שלושה תרגולים בלבד.
- משתמשים בפרויקט הקיים ולא פותחים פרויקט מרכזי נוסף.
- לכל תרגול יש תוצר אחד, Rubric אחד ונקודת סיום.
- כלי חדש יכול להחליף כלי בתרגול רק לאחר בדיקה; הוא לא מוסיף תרגול חדש.
- רעיונות נוספים נרשמים לגרסה עתידית ואינם נכנסים אוטומטית לגרסה 1.1.
- לאחר השלמת שלושת התרגולים עוצרים, בודקים את התוצאות ומחליטים בנפרד אם נדרש המשך.
