## מושגים מרכזיים

- API (Application Programming Interface): דרך מסודרת שבה מערכת מאפשרת למערכת אחרת לבקש ממנה מידע או לבצע פעולה.
- בקשה (Request): פנייה שמערכת אחת שולחת למערכת אחרת.
- תגובה (Response): התשובה שהמערכת מחזירה לבקשה.
- נקודת קצה (Endpoint): כתובת מסוימת ב-API שאליה שולחים בקשה.
- Payload: גוף המידע שנשלח בבקשה או מתקבל בתגובה, לרוב בפורמט JSON.
- Webhook: כתובת שמערכת אחת שולחת אליה מידע באופן אוטומטי כאשר אירוע מתרחש.
- Webhook URL: הכתובת שאליה נשלח המידע כאשר ה-Webhook מופעל.
- HTTP Method: סוג הפעולה בבקשת HTTP, למשל GET לקבלת מידע או POST לשליחת מידע.

## הסבר מרכזי

בשיעור הקודם הכרנו כלים כמו Make ו-n8n.

עכשיו נבין איך מערכות באמת מדברות זו עם זו.

כאשר אוטומציה מקבלת פנייה מטופס, שולחת אותה לכלי AI ושומרת תוצאה בגיליון, קורה דבר פשוט:

מערכות שולחות מידע זו לזו.

הדרך הנפוצה לעשות זאת היא דרך API או Webhook.

### מהו API

API הוא כמו דלפק שירות מסודר של מערכת.

מערכת אחת יכולה לפנות למערכת אחרת ולבקש:

- תן לי מידע.
- שמור מידע חדש.
- עדכן רשומה.
- מחק רשומה.
- הפעל פעולה.

לדוגמה:

מערכת האוטומציה שולחת בקשה לכלי AI:

"הנה טקסט של פנייה. החזר סיכום ושאלות המשך."

כלי ה-AI מחזיר תגובה:

"סיכום: הלקוח מתעניין בעיצוב סלון. חסרים פרטים: תקציב, מיקום ולוח זמנים."

### בקשה ותגובה

תקשורת עם API בנויה בדרך כלל מבקשה ותגובה.

בקשה כוללת:

- כתובת endpoint.
- סוג פעולה, למשל GET או POST.
- כותרות, למשל סוג המידע או הרשאות.
- payload, אם שולחים מידע.

תגובה כוללת:

- קוד מצב, למשל הצלחה או שגיאה.
- מידע שחזר מהמערכת.
- לפעמים הודעת שגיאה.

דוגמה פשוטה:

בקשה:

```text
POST https://api.example.com/leads
```

Payload:

```json
{
  "name": "דנה",
  "service": "עיצוב סלון",
  "message": "אני רוצה לבדוק אפשרות לעיצוב סלון בדירה חדשה"
}
```

תגובה:

```json
{
  "status": "received",
  "lead_id": "lead_123"
}
```

זו דוגמה לימודית בלבד. אין כאן מפתח אמיתי או מידע רגיש.

### מהו endpoint

Endpoint הוא כתובת מסוימת ב-API שמיועדת לפעולה מסוימת.

לדוגמה:

```text
https://api.example.com/leads
```

אפשר לחשוב על endpoint כמו דלת מסוימת במערכת.

דלת אחת לקבלת פניות.

דלת אחרת לעדכון לקוח.

דלת אחרת לשליחת הודעה.

### מהו payload

Payload הוא המידע עצמו שנשלח.

אם endpoint הוא הכתובת, ה-payload הוא החבילה.

לדוגמה, פנייה של לקוח יכולה לכלול:

- שם.
- אימייל.
- טלפון.
- סוג שירות.
- הודעה חופשית.

המידע הזה יישלח בדרך כלל בצורה מובנית, לרוב JSON. את JSON נלמד בשיעור הבא.

### מהו Webhook

Webhook הוא דרך שבה מערכת שולחת מידע למערכת אחרת כאשר משהו קורה.

API רגיל עובד הרבה פעמים כך:

"אני שואל את המערכת אם יש משהו חדש."

Webhook עובד כך:

"כאשר יש משהו חדש, המערכת תשלח אליי הודעה."

לדוגמה:

- לקוח שלח טופס.
- מערכת הטפסים שולחת את פרטי הפנייה ל-Webhook URL.
- Make או n8n מקבלים את המידע.
- האוטומציה מתחילה.

Webhook מתאים מאוד לטריגרים.

### Webhook URL

Webhook URL הוא הכתובת שאליה מערכת חיצונית שולחת מידע.

לדוגמה:

```text
https://automation.example.com/webhook/new-lead
```

זו כתובת דוגמה בלבד.

כאשר טופס חדש נשלח, מערכת הטפסים יכולה לשלוח לכתובת הזו את פרטי הפנייה.

### API מול Webhook

| שאלה | API | Webhook |
|---|---|---|
| מי מתחיל? | לרוב המערכת שמבקשת מידע | המערכת שבה קרה האירוע |
| מתאים ל | בקשת מידע או ביצוע פעולה | הפעלת אוטומציה כשקורה אירוע |
| דוגמה | שלח טקסט ל-AI וקבל סיכום | טופס חדש שולח פנייה לאוטומציה |
| חשיבה פשוטה | "אני מבקש" | "קרה משהו, אני מודיע" |

בפועל, אוטומציה יכולה להשתמש בשניהם:

Webhook מקבל פנייה חדשה.

API שולח את הפנייה ל-AI.

API נוסף שומר את התוצאה במערכת אחרת.

### מה חשוב לדעת על אבטחה

בשלב הזה לא נבנה מערכת אבטחה, אבל חשוב להבין עקרונות:

- לא מפרסמים API keys.
- לא שמים סודות בדוגמאות פתוחות.
- לא שולחים מידע רגיש בלי צורך.
- צריך לוודא שמי ששולח ל-Webhook באמת מורשה.
- צריך לבדוק מה עושים אם הבקשה נכשלת.

בדוגמאות בקורס נשתמש תמיד בכתובות דוגמה ובמידע לא רגיש.

## דוגמה מעשית

נמשיך עם סטודיו לעיצוב פנים.

תרחיש:

לקוח מילא טופס באתר. אנחנו רוצים להתחיל אוטומציה.

שלבים:

| שלב | מה קורה | סוג חיבור |
|---|---|---|
| 1 | לקוח שולח טופס | אירוע |
| 2 | הטופס שולח את פרטי הפנייה לכתובת Webhook | Webhook |
| 3 | האוטומציה מקבלת את הפנייה | Trigger |
| 4 | האוטומציה שולחת את טקסט הפנייה לכלי AI | API |
| 5 | כלי AI מחזיר סיכום ושאלות חסרות | API Response |
| 6 | האוטומציה שומרת את התוצאה בגיליון | API או חיבור מוכן |
| 7 | בעלת העסק מקבלת התראה לבדיקה | פעולה באוטומציה |

דוגמה ל-payload שנשלח מהטופס:

```json
{
  "customer_name": "דנה לוי",
  "email": "dana@example.com",
  "service_type": "עיצוב סלון",
  "message": "שלום, אני רוצה לעצב סלון בדירה חדשה ורוצה להבין איך מתחילים."
}
```

דוגמה לתגובה אפשרית מה-AI:

```json
{
  "summary": "הלקוחה מתעניינת בעיצוב סלון בדירה חדשה.",
  "missing_details": ["גודל החלל", "תקציב", "לוח זמנים"],
  "next_step": "לשלוח שאלות המשך לבדיקה ואישור של בעלת העסק."
}
```

שימו לב: אין כאן סודות, אין API key ואין מידע אמיתי של לקוח.

## טעויות נפוצות / נקודות זהירות

- לחשוב ש-API ו-Webhook הם אותו דבר.
- לשכוח ש-Webhook הוא כתובת שיכולה לקבל מידע ולכן צריך להגן עליה.
- לשלוח מידע רגיש ללא צורך.
- להשתמש בדוגמאות עם API key אמיתי.
- לא לבדוק מה קורה אם API מחזיר שגיאה.
- לא להגדיר מה ה-payload צריך לכלול.
- להניח שכל מערכת תומכת בכל פעולה.
- לבנות חיבור טכני לפני שמבינים את התהליך העסקי.

## חיבור לפרויקט המרכזי

בשיעור הזה נבין איך האוטומציה בפרויקט המרכזי תקבל ותעביר מידע בין מערכות.

משימה לפרויקט:

1. כתוב איזו מערכת שולחת את הפנייה הראשונה.
2. כתוב האם ההתחלה תהיה דרך Webhook או חיבור מוכן.
3. כתוב איזה payload צריך להגיע לאוטומציה.
4. כתוב לאיזו מערכת האוטומציה תשלח בקשת API.
5. כתוב מה התגובה הרצויה.
6. כתוב איזה מידע אסור לשלוח בשלב הראשון.

דוגמה:

- מקור: טופס באתר.
- התחלה: Webhook שמקבל פנייה חדשה.
- Payload: שם, אימייל, סוג שירות והודעה.
- API: שליחת ההודעה לכלי AI לסיכום.
- תגובה: סיכום, פרטים חסרים ושאלות המשך.
- לא שולחים: מידע רגיש או מפתח API בדוגמה.

## סיכום קצר

בשיעור הזה למדנו איך מערכות מדברות זו עם זו באמצעות APIs ו-Webhooks.
