## מטרת השיעור

מטרת השיעור היא להבין איך אפליקציית AI בנויה ברמה כללית: אילו חלקים יש במערכת, מה התפקיד של כל חלק, ואיך בקשה אחת עוברת מהמשתמש עד לתשובה.

עד עכשיו למדנו על אוטומציות, נתונים, בסיס ידע, RAG וסוכני AI. בשיעור הזה נחבר את החלקים לתמונה אחת: איך נראית מערכת AI עסקית בסיסית.

## מה הלומד ידע בסוף השיעור

בסוף השיעור הלומד ידע:

1. להסביר מהם Frontend, Backend, API, Database, AI Model, Tools ו-Logs.
2. לתאר איך בקשה עוברת באפליקציית AI.
3. להבין איפה נשמר מידע ואיפה מופעל מודל AI.
4. לזהות אילו חלקים נדרשים במערכת המרכזית של הקורס.
5. להבדיל בין מבנה מערכת לבין מימוש טכני מלא.

אין צורך לדעת לתכנת אפליקציה מלאה.

## מושגים מרכזיים

- צד לקוח (Frontend): החלק שהמשתמש רואה ומפעיל, למשל טופס פנייה או מסך ניהול.
- צד שרת (Backend): החלק שמטפל בלוגיקה מאחורי הקלעים, למשל שמירה, קריאה למודל AI והפעלת כלים.
- ממשק API (Application Programming Interface): הדרך שבה חלקים במערכת שולחים בקשות ומקבלים תשובות.
- מסד נתונים (Database): המקום שבו נשמר מידע מסודר, למשל פניות לקוחות וסטטוסים.
- מודל AI (AI Model): המודל שמנתח טקסט, מסכם, מסווג או מנסח תשובה.
- כלים (Tools): פעולות שהמערכת או הסוכן יכולים להפעיל, כמו שליפה מבסיס ידע או עדכון רשומה.
- לוגים (Logs): תיעוד של מה קרה במערכת, כדי להבין תקלות, לבדוק זרימות ולשפר איכות.

## הסבר מרכזי

אפליקציית AI היא לא רק "מודל AI".

המודל הוא חלק חשוב, אבל סביבו יש מערכת שלמה:

- מסך שבו המשתמש מכניס מידע.
- שרת שמקבל את הבקשה.
- בסיס נתונים ששומר מידע.
- קריאה למודל AI.
- כלים שהמודל או הסוכן יכולים להפעיל.
- לוגים שעוזרים להבין מה קרה.
- לפעמים גם אישור אדם לפני פעולה רגישה.

אפשר לחשוב על זה כמו מסעדה:

המשתמש רואה את המלצר ואת התפריט. זה ה-Frontend.  
המטבח מכין את המנה. זה ה-Backend.  
ההזמנה עוברת למטבח בצורה מסודרת. זה ה-API.  
מחברת ההזמנות שומרת מה הוזמן. זה ה-Database.  
הטבח שמציע ניסוח או החלטה הוא ה-AI Model.  
כלים נוספים עוזרים, למשל בדיקת מלאי או חישוב מחיר.  
לוגים הם התיעוד של מה קרה אם משהו השתבש.

### החלקים המרכזיים באפליקציית AI

| חלק | תפקיד פשוט |
|---|---|
| Frontend | מציג למשתמש טופס, כפתורים ותוצאות |
| Backend | מקבל בקשות ומנהל את הלוגיקה |
| API | מחבר בין חלקים שונים במערכת |
| Database | שומר פניות, סטטוסים ותוצאות |
| AI Model | מנתח, מסכם, מסווג או מנסח |
| Tools | מאפשרים פעולות כמו שליפה ועדכון |
| Logs | מתעדים מה קרה לצורך בדיקה ושיפור |

### איך בקשה עוברת במערכת

ניקח דוגמה פשוטה: לקוח שולח פנייה דרך טופס.

הזרימה יכולה להיות:

1. הלקוח ממלא טופס ב-Frontend.
2. ה-Frontend שולח בקשה ל-Backend.
3. ה-Backend בודק שהמידע בסיסי ותקין.
4. ה-Backend שומר רשומה חדשה ב-Database.
5. ה-Backend שולח את תוכן הפנייה ל-AI Model.
6. המודל מחזיר סיכום, סוג שירות ופרטים חסרים.
7. ה-Backend מעדכן את הרשומה עם תוצאת ה-AI.
8. אם צריך, המערכת שולפת מידע מבסיס הידע.
9. הסוכן מכין טיוטת תשובה.
10. המערכת שומרת לוגים של התהליך.
11. אם יש פעולה רגישה, אדם מאשר לפני שליחה.

שימו לב: כל חלק עושה תפקיד מוגדר. זה עוזר לבנות מערכת שאפשר להבין, לבדוק ולתקן.

### למה לא לשים הכל במקום אחד

מתחילים לפעמים חושבים:

"נשלח הכל ישר למודל AI והוא כבר יסתדר."

זו גישה מסוכנת.

אם אין Backend ברור, קשה לדעת:

- מה נשמר.
- מה נשלח למודל.
- מי אישר פעולה.
- למה התקבלה תשובה מסוימת.
- איפה הייתה התקלה.
- איך לשפר את התהליך.

מערכת טובה מפרידה בין אחריות:

- ה-Frontend אחראי על ממשק.
- ה-Backend אחראי על זרימה ולוגיקה.
- ה-Database אחראי על שמירת מידע.
- ה-AI אחראי על ניתוח וניסוח.
- הכלים אחראים על פעולות מוגדרות.
- הלוגים אחראים על תיעוד.

### איפה נכנס הסוכן

סוכן AI יכול לשבת בתוך ה-Backend או להיות חלק מתהליך שמופעל על ידי ה-Backend.

לדוגמה:

ה-Backend מקבל פנייה חדשה ומפעיל סוכן עם מטרה:

"נתח את הפנייה, שלוף מידע רלוונטי, עדכן סטטוס והכן טיוטת תשובה."

הסוכן לא מחליף את כל המערכת. הוא פועל בתוך מערכת עם גבולות.

### מה חשוב לא לשמור או להציג

גם באפליקציית AI פשוטה צריך להיזהר ממידע רגיש.

לא מציגים בדוגמאות ולא שומרים ללא צורך:

- סיסמאות.
- מפתחות API.
- פרטי אשראי.
- מידע אישי של לקוחות אמיתיים.
- הערות פנימיות רגישות.

אם מידע לא נחוץ לתהליך, עדיף לא לאסוף אותו.

## דוגמה מעשית

נמשיך עם סטודיו לעיצוב פנים.

המערכת המרכזית יכולה להיראות כך:

| חלק במערכת | דוגמה בפרויקט |
|---|---|
| Frontend | טופס פנייה ללקוח ומסך פניות לבעלת העסק |
| Backend | קבלת הפנייה, הפעלת AI ועדכון סטטוס |
| API | בקשה מהטופס לשרת, ובקשה מהשרת למודל AI |
| Database | טבלת פניות לקוחות |
| AI Model | סיכום הפנייה וזיהוי פרטים חסרים |
| Tools | שליפת בסיס ידע ועדכון רשומה |
| Logs | תיעוד שהפנייה נשמרה, נותחה ונשלחה לאישור |

דוגמת זרימה:

1. לקוח כותב: "אני רוצה לעצב סלון בדירה חדשה."
2. הטופס שולח את הפנייה לשרת.
3. השרת שומר פנייה חדשה בסטטוס "חדש".
4. המודל מסכם: "הלקוח מתעניין בעיצוב סלון."
5. המערכת מזהה שחסרים תקציב, גודל חלל ולוח זמנים.
6. הסטטוס מתעדכן ל"חסר מידע".
7. נשלפת תשובה מומלצת מבסיס הידע.
8. נוצרת טיוטת תשובה לאישור בעלת העסק.
9. הלוגים מתעדים את השלבים.

בשלב הזה לא צריך לבנות את כל הקוד. צריך להבין את המבנה.

## טעויות נפוצות / נקודות זהירות

- לחשוב שמודל AI הוא כל האפליקציה.
- לשמור מידע בלי להבין למה הוא נחוץ.
- לשלוח למודל יותר מדי מידע במקום רק את מה שרלוונטי.
- לא לשמור לוגים ולכן לא להבין מה קרה בתקלה.
- לערבב בין Frontend לבין Backend.
- לתת לסוכן לבצע פעולה רגישה בלי אישור.
- לא להפריד בין טיוטה לבין הודעה שנשלחה בפועל.
- להשתמש בדוגמאות עם מידע אמיתי או רגיש.

מבנה ברור עוזר לבנות פתרון שאפשר לבדוק, לתחזק ולהסביר ללקוח.

## סיכום קצר

אפליקציית AI מורכבת מכמה חלקים, ולא רק ממודל AI.

Frontend מציג את הממשק, Backend מנהל את הלוגיקה, API מחבר בין חלקים, Database שומר מידע, AI Model מנתח ומנסח, Tools מבצעים פעולות מוגדרות, ו-Logs מתעדים מה קרה.

בשיעור הבא נלמד איך לעבוד בצורה מסודרת עם כלי פיתוח AI כמו Codex ו-Claude Code, יחד עם GitHub, בלי לוותר על הבנה, בדיקה ואחריות.
