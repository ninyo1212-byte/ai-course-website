## מטרת השיעור

מטרת השיעור היא להבין איך לעבוד עם כלי פיתוח AI בצורה מסודרת ובטוחה: להגדיר משימה קטנה, לבקש Plan, לבצע שינוי, לבדוק אותו, ולשמור אותו ב-GitHub.

השיעור לא מלמד התקנה של כלי פיתוח ולא נכנס לקוד עמוק. הדגש הוא על תהליך עבודה נכון ועל אחריות אנושית בזמן שימוש בכלי קוד חכמים.

## מה הלומד ידע בסוף השיעור

בסוף השיעור הלומד ידע:

1. להסביר איך Codex ו-Claude Code יכולים לעזור בפיתוח.
2. להבין למה צריך לעבוד במשימות קטנות וברורות.
3. לתאר תהליך עבודה בסיסי: אפיון, משימה, Plan, ביצוע, בדיקה ו-commit.
4. להסביר מה תפקיד GitHub בתיעוד ושמירת שינויים.
5. לזהות סיכונים בעבודה עם כלי פיתוח AI.

## ידע מקדים נדרש

נדרש ידע משיעורים 1-30 בלבד:

- מהו פתרון AI עסקי.
- מהו סוכן AI ומהם כלים והרשאות.

- איך בנויה אפליקציית AI ברמה כללית.
- למה חשוב לבדוק תוצאה ולא לסמוך על AI בעיניים עצומות.

אין צורך לדעת לעבוד לעומק עם Git או לכתוב קוד מתקדם.

## מושגים מרכזיים

- כלי פיתוח AI (AI Coding Tool): כלי שעוזר להבין קוד, להציע שינויים, לכתוב קוד, לתקן בעיות או ליצור בדיקות.
- Codex: כלי פיתוח מבוסס AI של OpenAI שיכול לעזור במשימות קוד, בדיקה, הסבר ושינוי קבצים.
- Claude Code: כלי פיתוח מבוסס Claude שיכול לעבוד עם קבצים, להבין קוד, להציע שינויים ולסייע בתהליכי פיתוח.
- GitHub: פלטפורמה לניהול קוד, היסטוריית שינויים, שיתוף עבודה וסקירת שינויים.
- קומיט (Commit): שמירה מתועדת של שינוי בקוד.
- בקשת משיכה / בקשת סקירה (Pull Request): בקשה לסקירה ואישור של שינוי לפני שילובו בענף המרכזי.
- תוכנית פעולה (Plan): תוכנית קצרה שמסבירה מה הכלי עומד לעשות לפני שהוא משנה קבצים.

## הסבר מרכזי

כלי פיתוח AI יכולים לחסוך זמן, לעזור להבין קוד ולהציע פתרונות.

אבל הם לא מחליפים הבנה, בדיקה ואחריות.

כלי כזה יכול:

- להסביר מבנה של קוד.
- למצוא קבצים רלוונטיים.
- להציע פתרון לבעיה.
- לכתוב קטע קוד.
- להוסיף בדיקות.
- לעזור לנסח commit או Pull Request.

אבל הוא גם יכול:

- לשנות קובץ לא נכון.
- למחוק לוגיקה חשובה.
- להציע פתרון שנראה טוב אבל לא עובד.
- להתעלם ממקרי קצה.
- לחשוף מידע אם נותנים לו סודות או נתונים אמיתיים.
- לבצע שינוי גדול מדי שקשה לבדוק.

לכן צריך לעבוד איתו בתהליך מסודר.

### תהליך עבודה מומלץ

תהליך עבודה פשוט עם Codex, Claude Code או כלי דומה:

1. אפיון קצר.
2. משימה קטנה.
3. Plan לפני ביצוע.
4. ביצוע שינוי מוגבל.
5. בדיקה.
6. סקירת השינוי.
7. Commit עם תיאור ברור.
8. Pull Request אם עובדים בצוות או רוצים סקירה.

### 1. אפיון קצר

לפני שמבקשים מהכלי לכתוב קוד, צריך להבין מה רוצים.

לא טוב:

"תבנה לי מערכת פניות."

טוב יותר:

"הוסף שדה `status` לפנייה חדשה, כך שכל פנייה תתחיל בסטטוס 'חדש'. אל תשנה חלקים אחרים."

אפיון קצר צריך לכלול:

- מה הבעיה.
- מה השינוי הרצוי.
- מה לא לשנות.
- איך נדע שזה עובד.

### 2. משימה קטנה

כלי פיתוח AI עובד טוב יותר כשנותנים לו משימה מוגבלת.

משימה גדולה:

"בנה את כל הסוכן העסקי."

משימה קטנה:

"הוסף בדיקה שמונעת שליחת תשובה אם `needs_human_review` הוא true."

משימה קטנה קלה יותר להבנה, לביצוע ולבדיקה.

### 3. Plan לפני ביצוע

לפני שינוי קבצים, מבקשים מהכלי להסביר:

- אילו קבצים הוא יקרא.
- אילו קבצים הוא ישנה.
- מה השינוי הצפוי.
- איך הוא יבדוק את עצמו.

אם התוכנית לא ברורה, מתקנים אותה לפני ביצוע.

זה חשוב במיוחד כי כלי פיתוח AI יכולים לפעול מהר. מהירות בלי בקרה יכולה ליצור נזק.

### 4. ביצוע שינוי מוגבל

אחרי אישור התוכנית, נותנים לכלי לבצע רק את המשימה שאושרה.

אם הכלי מזהה רעיון נוסף, הוא צריך להציע אותו ולא לבצע לבד.

דוגמה:

אם ביקשנו להוסיף בדיקת סטטוס, אין סיבה שהכלי ישנה עיצוב, מבנה תיקיות או שם של פונקציות אחרות.

### 5. בדיקה

לא מספיק שהכלי אומר "סיימתי".

צריך לבדוק:

- האם השינוי באמת בוצע.
- האם חלקים קיימים עדיין עובדים.
- האם יש שגיאות.
- האם מקרי קצה נבדקו.
- האם לא נחשף מידע רגיש.

אם יש בדיקות אוטומטיות, מריצים אותן. אם אין, עושים בדיקה ידנית לפי תרחישים פשוטים.

### 6. Commit

Commit הוא שמירה מתועדת של שינוי.

Commit טוב כולל שינוי קטן וברור.

דוגמה לשם commit:

```text
add human review guard before sending reply
```

או בעברית בפרויקט לימודי:

```text
הוסף עצירת אישור אדם לפני שליחת תשובה
```

העיקר שהשם יסביר מה השתנה.

### 7. GitHub וסקירה

GitHub עוזר לשמור היסטוריה:

- מה השתנה.
- מי שינה.
- מתי.
- למה.
- אילו בדיקות עברו.
- אילו הערות התקבלו בסקירה.

אם עובדים לבד, GitHub עדיין עוזר לשמור גרסאות ולחזור אחורה במקרה של טעות.

אם עובדים בצוות, Pull Request מאפשר לבקש סקירה לפני שמכניסים שינוי למערכת הראשית.

### כלל חשוב

לא נותנים לכלי קוד גישה חופשית לשנות הכל.

עובדים כך:

משימה קטנה -> Plan -> אישור -> שינוי -> בדיקה -> commit.

## דוגמה מעשית

נמשיך עם סטודיו לעיצוב פנים.

רוצים להוסיף למערכת כלל:

"אם הפנייה דורשת בדיקה אנושית, המערכת לא תשלח תשובה אוטומטית."

בקשה טובה לכלי פיתוח AI:

```text
בדוק איפה במערכת נוצרת טיוטת תשובה לפנייה.
הכן Plan קצר לפני שינוי.
המטרה: אם needs_human_review = true, המערכת תיצור טיוטה בלבד ולא תשלח אותה.
אל תשנה את מבנה בסיס הנתונים ואל תשנה עיצוב.
בסוף הצע דרך בדיקה פשוטה.
```

Plan צפוי:

1. לאתר את הקובץ שמטפל בשליחת תשובה.
2. לבדוק איפה נקרא השדה `needs_human_review`.
3. להוסיף תנאי לפני שליחה.
4. להחזיר סטטוס "ממתין לאישור".
5. לבדוק עם פנייה לדוגמה.

בדיקה ידנית:

| מצב | ציפייה |
|---|---|
| `needs_human_review = true` | נוצרת טיוטה בלבד |
| `needs_human_review = false` | אפשר להמשיך לשליחה לפי הכללים |
| השדה חסר | המערכת עוצרת לבדיקה ולא מנחשת |

Commit אפשרי:

```text
add guard for human review before customer reply
```

## טעויות נפוצות / נקודות זהירות

- לתת לכלי משימה גדולה מדי.
- לדלג על Plan.
- לא לקרוא את השינוי לפני שמאשרים.
- לא להריץ בדיקות.
- לעשות commit עם הרבה שינויים לא קשורים.
- לתת לכלי לשנות קבצים שלא קשורים למשימה.
- להכניס סיסמאות, מפתחות API או נתוני לקוחות אמיתיים לשיחה.
- להניח שהקוד נכון רק כי הוא נראה מסודר.
- לא לשמור גרסה לפני שינוי משמעותי.

כלי פיתוח AI הם עוזרים חזקים, אבל המשתמש עדיין אחראי להבין מה משתנה ולבדוק את התוצאה.

בפרויקט המרכזי נשתמש בכלי פיתוח AI כדי לבנות או לשפר חלקים קטנים במערכת.

לדוגמה:

- להוסיף שדה חדש לפנייה.
- להוסיף בדיקת אישור אדם.
- לשפר הודעת שגיאה.
- להוסיף לוג.
- לכתוב בדיקה פשוטה.
- להסביר קובץ קיים.

בכל פעם נעבוד באותו תהליך:

אפיון -> משימה קטנה -> Plan -> ביצוע -> בדיקה -> commit.

כך אפשר להתקדם מהר, אבל בלי לאבד שליטה.
