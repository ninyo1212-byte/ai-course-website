2. לתאר איך אוטומציה שומרת פנייה חדשה ומעדכנת את הסטטוס שלה.
3. לתכנן פעולות בסיסיות של חיפוש ושליפת מידע רלוונטי על פנייה.

## מושגים מרכזיים

- יצירת רשומה (Create / Insert): הוספת פריט מידע חדש לטבלה, למשל פנייה חדשה.
- עדכון רשומה (Update): שינוי מידע ברשומה קיימת, למשל שינוי סטטוס.
- שליפה (Read / Retrieve): קבלת מידע שנשמר, למשל חיפוש פנייה לפי מזהה.
- חיפוש (Search / Query): בקשה למצוא רשומות לפי תנאי, למשל כל הפניות בסטטוס "חסר מידע".
- מחיקה (Delete): הסרת רשומה. בשלב זה נזכיר את המושג בלבד ולא נשתמש בו כדבר ראשון.
- CRUD: קיצור לפעולות בסיסיות על מידע: Create, Read, Update, Delete.
- מסנן (Filter): תנאי שמצמצם אילו רשומות יוחזרו.
- סטטוס (Status): שדה שמראה באיזה מצב נמצאת פנייה בתהליך.

## הסבר מרכזי

בשיעור הקודם תכננו טבלה בסיסית לפניות לקוחות.

עכשיו נבין מה המערכת עושה עם הטבלה הזו בפועל.

ברוב המערכות העסקיות יש ארבע פעולות בסיסיות על מידע:

1. שמירה של מידע חדש.
2. שליפה של מידע קיים.
3. עדכון של מידע קיים.
4. לפעמים מחיקה של מידע.

פעולות אלה נקראות לעיתים CRUD.

בשלב הזה נתמקד בשלוש פעולות חשובות לפרויקט שלנו:

- שמירה.
- עדכון.
- שליפה.

### שמירה: יצירת רשומה חדשה

כאשר לקוח שולח פנייה, המערכת צריכה לשמור אותה.

לדוגמה:

```json
{
  "lead_id": "lead_001",
  "customer_name": "דנה לוי",
  "service_type": "עיצוב סלון",
  "original_message": "אני רוצה לעצב סלון בדירה חדשה",
  "status": "חדש",
  "created_at": "2026-07-09"
}
```

זו יצירת רשומה חדשה בטבלת הפניות.

המטרה היא שלא נאבד את הפנייה ושיהיה לה מקום קבוע.

### מה חשוב לשמור בפנייה חדשה

בדרך כלל נשמור:

- מזהה ייחודי.
- פרטי לקוח בסיסיים.
- מקור הפנייה.
- טקסט הפנייה המקורי.
- סטטוס ראשוני.
- תאריך יצירה.

אחרי שה-AI יעבד את הפנייה, אפשר לשמור גם:

- סיכום AI.
- פרטים חסרים.
- שאלות המשך.
- האם צריך בדיקה אנושית.

### עדכון: שינוי רשומה קיימת

מידע עסקי משתנה.

פנייה יכולה להתחיל בסטטוס "חדש", ואז לעבור ל:

- חסר מידע.
- בטיפול.
- ממתין ללקוח.
- מתאים לשיחה.
- נסגר.

עדכון רשומה אומר:

"מצא את הפנייה הקיימת ושנה בה שדה אחד או יותר."

לדוגמה:

לפני:

```json
{
  "lead_id": "lead_001",
  "status": "חדש"
}
```

אחרי:

```json
{
  "lead_id": "lead_001",
  "status": "חסר מידע",
  "missing_details": ["תקציב", "גודל חלל"],
  "updated_at": "2026-07-09"
}
```

שימו לב: אנחנו לא יוצרים פנייה חדשה. אנחנו מעדכנים את אותה פנייה לפי `lead_id`.

### למה מזהה חשוב בעדכון

כדי לעדכן נכון, צריך לדעת איזו רשומה לעדכן.

אם נעדכן לפי שם לקוח בלבד, ייתכן שיש שני לקוחות באותו שם.

אם נעדכן לפי הודעה, ייתכן שיש הודעות דומות.

לכן משתמשים במזהה כמו `lead_id`.

המזהה מאפשר לומר למערכת:

"עדכני את הפנייה הזו, לא פנייה אחרת."

### שליפה: קבלת מידע קיים

שליפה היא קבלת מידע שכבר נשמר.

לדוגמה:

- מצא פנייה לפי `lead_id`.
- הצג את כל הפניות החדשות.
- הצג פניות שחסר בהן מידע.
- הצג פניות שנוצרו השבוע.
- מצא את כל הפניות של לקוח מסוים.

שליפה מאפשרת לאדם או לאוטומציה להמשיך תהליך בלי להתחיל מאפס.

### חיפוש לפי תנאי

לעיתים לא נרצה רשומה אחת, אלא רשימה של רשומות שמתאימות לתנאי.

לדוגמה:

```text
הצג את כל הפניות שבהן status = "חסר מידע"
```

או:

```text
הצג את כל הפניות שבהן needs_human_review = true
```

זה חיפוש עם מסנן.

מסנן עוזר לשלוף רק את המידע הרלוונטי.

### שימוש עסקי בשליפה

שליפה עוזרת בתהליכים כמו:

- לדעת על מי צריך לחזור היום.
- לבדוק אילו פניות מחכות ללקוח.
- להציג לבעל העסק פנייה מלאה לפני תגובה.
- לשלוח ל-AI את ההקשר הרלוונטי.
- למדוד כמה פניות נמצאות בכל סטטוס.

חשוב: בשלב הזה אנחנו לא נכנסים עדיין ל-RAG. כלומר, אנחנו לא בונים מנגנון של שליפת ידע מורכב כדי לשפר תשובות AI.

אנחנו רק לומדים איך מידע עסקי נשמר ונשלף בצורה בסיסית.

### דוגמת זרימה מלאה

אוטומציה לטיפול בפנייה:

1. לקוח שולח טופס.
2. המערכת יוצרת רשומה חדשה בטבלת `leads`.
3. AI מסכם את הפנייה.
4. המערכת מעדכנת את אותה רשומה עם הסיכום והפרטים החסרים.
5. אם חסר מידע, הסטטוס מתעדכן ל"חסר מידע".
6. בעל העסק שולף את הפנייה כדי לבדוק מה לשלוח ללקוח.

כל זה נשען על שלוש פעולות:

- יצירה.
- עדכון.
- שליפה.

### מה חשוב לא לעשות

לא כל שינוי צריך ליצור רשומה חדשה.

אם כל עדכון יוצר פנייה חדשה, יהיו כפילויות.

לדוגמה:

לא טוב:

- פנייה חדשה נוצרת.
- AI מסכם אותה ויוצר פנייה נוספת.
- בעל העסק מעדכן סטטוס ונוצרת פנייה שלישית.

טוב:

- פנייה אחת נוצרת.
- הסיכום מתווסף לאותה פנייה.
- הסטטוס מתעדכן באותה פנייה.

כך נשמרת תמונה ברורה.

## דוגמה מעשית

נמשיך עם סטודיו לעיצוב פנים.

פנייה חדשה מתקבלת:

```json
{
  "lead_id": "lead_001",
  "customer_name": "דנה לוי",
  "service_type": "עיצוב סלון",
  "original_message": "שלום, אני רוצה לעצב סלון בדירה חדשה.",
  "status": "חדש",
  "created_at": "2026-07-09"
}
```

אחרי סיכום AI, המערכת מעדכנת:

```json
{
  "lead_id": "lead_001",
  "ai_summary": "הלקוחה מתעניינת בעיצוב סלון בדירה חדשה.",
  "missing_details": ["תקציב", "גודל החלל", "לוח זמנים"],
  "status": "חסר מידע",
  "needs_human_review": true,
  "updated_at": "2026-07-09"
}
```

לאחר מכן בעלת העסק יכולה לשלוף:

```text
כל הפניות שבהן status = "חסר מידע"
```

ולקבל רשימה של פניות שצריך להשלים בהן פרטים.

בשלב הזה המערכת לא שולחת הודעה לבד. היא רק עוזרת לשמור, לעדכן ולמצוא מידע.

## טעויות נפוצות / נקודות זהירות

- ליצור רשומה חדשה במקום לעדכן רשומה קיימת.
- לעדכן לפי שם לקוח במקום לפי מזהה ייחודי.
- לא לשמור סטטוס.
- לא לעדכן `updated_at` או תיעוד אחר של שינוי.
- לשלוף יותר מדי מידע במקום רק מה שצריך.
- לשמור מידע רגיש ללא צורך.
- למחוק מידע לפני שמבינים אם צריך לשמור היסטוריה.
- לא להגדיר מה קורה אם לא נמצאה רשומה לעדכון.

## חיבור לפרויקט המרכזי

בשיעור הזה נגדיר איך המערכת בפרויקט המרכזי תשמור, תעדכן ותשלוף מידע על פניות.

משימה לפרויקט:

1. כתוב מה קורה כאשר פנייה חדשה נכנסת.
2. כתוב אילו שדות נשמרים ביצירת הרשומה.
3. כתוב אילו שדות מתעדכנים אחרי פעולת AI.
4. כתוב אילו חיפושים בסיסיים המערכת צריכה לתמוך בהם.
5. כתוב מה לא נשמר כדי להימנע ממידע רגיש או מיותר.

דוגמה:

- יצירה: פנייה חדשה נשמרת עם סטטוס "חדש".
- עדכון: AI מוסיף סיכום, פרטים חסרים וסטטוס "חסר מידע".
- שליפה: בעלת העסק שולפת את כל הפניות שדורשות בדיקה אנושית.

## סיכום קצר

בשיעור הזה למדנו איך מערכת שומרת, מעדכנת ושולפת מידע עסקי.
