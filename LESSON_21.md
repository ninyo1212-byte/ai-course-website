## מושגים מרכזיים

- JSON (JavaScript Object Notation): פורמט טקסט נפוץ להעברת מידע מובנה בין מערכות.
- מפתח/ערך (Key/Value): זוג שבו מפתח מתאר את שם השדה והערך הוא המידע עצמו.
- אובייקט (Object): קבוצה של שדות בתוך סוגריים מסולסלים.
- מערך (Array): רשימה של ערכים בתוך סוגריים מרובעים.
- מחרוזת (String): טקסט, תמיד בתוך מרכאות כפולות ב-JSON.
- מספר (Number): ערך מספרי, בלי מרכאות.
- בוליאני (Boolean): ערך נכון או לא נכון: true או false.
- null: ערך שמציין שאין מידע או שהשדה ריק.

## הסבר מרכזי

בשיעור הקודם למדנו ש-API ו-Webhook מעבירים מידע בין מערכות.

עכשיו נלמד איך המידע הזה נראה בפועל.

אחד הפורמטים הנפוצים ביותר להעברת מידע בין מערכות הוא JSON.

JSON הוא טקסט מסודר שמאפשר למערכות להבין:

- מה שם הלקוח.
- מה האימייל שלו.
- איזה שירות הוא ביקש.
- מה כתוב בפנייה.
- האם חסרים פרטים.
- אילו שאלות המשך צריך לשאול.

### למה צריך JSON

אדם יכול לקרוא הודעה חופשית כמו:

"שלום, אני דנה. אני רוצה לעצב סלון בדירה חדשה."

אבל מערכת צריכה מידע מסודר יותר.

לדוגמה:

```json
{
  "customer_name": "דנה",
  "service_type": "עיצוב סלון",
  "message": "אני רוצה לעצב סלון בדירה חדשה"
}
```

כאן כל חלק קיבל שם ברור.

המערכת יודעת שהשם נמצא ב-`customer_name`, שסוג השירות נמצא ב-`service_type`, ושההודעה נמצאת ב-`message`.

### Key/Value

הבסיס של JSON הוא זוגות של מפתח וערך.

לדוגמה:

```json
{
  "customer_name": "דנה"
}
```

המפתח הוא:

```text
customer_name
```

הערך הוא:

```text
דנה
```

המפתח מתאר מה זה המידע.

הערך הוא המידע עצמו.

### Object

Object הוא קבוצה של שדות.

לדוגמה:

```json
{
  "customer_name": "דנה",
  "email": "dana@example.com",
  "service_type": "עיצוב סלון"
}
```

האובייקט מתחיל ב-`{` ונגמר ב-`}`.

בפנים יש שדות, וכל שדה בנוי ממפתח וערך.

### Array

Array הוא רשימה.

לדוגמה:

```json
{
  "missing_details": ["תקציב", "מיקום", "לוח זמנים"]
}
```

המערך מתחיל ב-`[` ונגמר ב-`]`.

כאן יש רשימה של פרטים חסרים.

אפשר להשתמש במערך כאשר יש יותר מפריט אחד מאותו סוג:

- שאלות המשך.
- פרטים חסרים.
- קבצים.
- מוצרים.
- תגיות.

### סוגי ערכים בסיסיים

ב-JSON יש כמה סוגי ערכים נפוצים.

#### 1. String

טקסט בתוך מרכאות כפולות:

```json
{
  "status": "חסר מידע"
}
```

#### 2. Number

מספר בלי מרכאות:

```json
{
  "estimated_budget": 15000
}
```

#### 3. Boolean

נכון או לא נכון:

```json
{
  "needs_human_review": true
}
```

#### 4. Object בתוך Object

אובייקט יכול להכיל אובייקט אחר:

```json
{
  "customer": {
    "name": "דנה",
    "email": "dana@example.com"
  }
}
```

#### 5. Array של Objects

רשימה יכולה להכיל אובייקטים:

```json
{
  "questions": [
    {
      "field": "budget",
      "question": "מה התקציב המשוער?"
    },
    {
      "field": "timeline",
      "question": "מתי תרצי להתחיל?"
    }
  ]
}
```

זה שימושי כאשר רוצים להעביר רשימה מסודרת של פריטים, וכל פריט כולל כמה שדות.

### דוגמה מלאה: פנייה של לקוח

פנייה יכולה לעבור בין מערכות כך:

```json
{
  "lead_id": "lead_123",
  "customer": {
    "name": "דנה לוי",
    "email": "dana@example.com",
    "phone": "050-0000000"
  },
  "service_type": "עיצוב סלון",
  "message": "שלום, אני רוצה לעצב סלון בדירה חדשה ורוצה להבין איך מתחילים.",
  "source": "website_form",
  "urgent": false
}
```

מה רואים כאן:

- `lead_id` הוא מזהה פנייה.
- `customer` הוא אובייקט עם פרטי הלקוח.
- `service_type` הוא סוג השירות.
- `message` הוא הטקסט החופשי.
- `source` הוא מקור הפנייה.
- `urgent` הוא ערך boolean.

### דוגמה לפלט מ-AI

אחרי שה-AI מסכם את הפנייה, הוא יכול להחזיר מידע מובנה:

```json
{
  "summary": "הלקוחה מתעניינת בעיצוב סלון בדירה חדשה.",
  "existing_details": {
    "space_type": "סלון",
    "property_type": "דירה חדשה"
  },
  "missing_details": ["תקציב", "גודל החלל", "לוח זמנים"],
  "recommended_questions": [
    "מה התקציב המשוער לפרויקט?",
    "מה גודל הסלון?",
    "מתי תרצי להתחיל?"
  ],
  "needs_human_review": true
}
```

כשהפלט מסודר כך, האוטומציה יכולה להשתמש בו:

- לשמור סיכום בשדה אחד.
- לשמור פרטים חסרים בשדה אחר.
- לשלוח לבעל העסק שאלות לבדיקה.
- לסמן שהפנייה דורשת אישור אדם.

### כללים בסיסיים ל-JSON תקין

חשוב לשמור על כמה כללים:

- מפתחות כתובים במרכאות כפולות.
- טקסט כתוב במרכאות כפולות.
- בין שדות יש פסיק.
- אין פסיק אחרי השדה האחרון.
- אובייקטים משתמשים ב-`{}`.
- מערכים משתמשים ב-`[]`.
- לא כותבים הערות בתוך JSON.
- לא מכניסים סודות כמו API keys בדוגמאות.

דוגמה לא תקינה:

```json
{
  "name": "דנה",
  "service": "עיצוב סלון",
}
```

הבעיה: יש פסיק אחרי השדה האחרון.

דוגמה תקינה:

```json
{
  "name": "דנה",
  "service": "עיצוב סלון"
}
```

### למה JSON חשוב באוטומציות

Make, n8n, APIs ו-Webhooks עובדים הרבה פעמים עם מידע מובנה.

גם אם הכלי מציג את זה בצורה ויזואלית, מאחורי הקלעים המידע נראה לעיתים כמו JSON.

אם מבינים JSON, קל יותר:

- להבין מה התקבל מ-Webhook.
- לדעת איזה שדה לשלוח ל-AI.
- לשמור את התשובה במקום הנכון.
- לבדוק למה פעולה נכשלה.
- להעביר מידע בין מערכות בלי בלבול.

לא צריך להיות מתכנת כדי לקרוא JSON בסיסי. צריך להבין את המבנה.

## דוגמה מעשית

נמשיך עם סטודיו לעיצוב פנים.

הטופס באתר שולח פנייה לאוטומציה:

```json
{
  "customer": {
    "name": "דנה לוי",
    "email": "dana@example.com"
  },
  "service_type": "עיצוב סלון",
  "message": "אני רוצה לעצב סלון בדירה חדשה. אשמח להבין תהליך ועלויות.",
  "source": "website_form"
}
```

האוטומציה שולחת ל-AI את `message` ואת `service_type`.

ה-AI מחזיר:

```json
{
  "summary": "הלקוחה רוצה להבין תהליך ועלויות לעיצוב סלון בדירה חדשה.",
  "missing_details": ["גודל החלל", "תקציב", "לוח זמנים", "סגנון רצוי"],
  "recommended_questions": [
    "מה גודל הסלון?",
    "מה התקציב המשוער?",
    "מתי תרצי להתחיל?",
    "איזה סגנון עיצוב את אוהבת?"
  ],
  "status": "חסר מידע",
  "needs_human_review": true
}
```

עכשיו אפשר להעביר כל שדה למקום מתאים:

| שדה JSON | שימוש באוטומציה |
|---|---|
| `summary` | שמירה בשדה סיכום |
| `missing_details` | הצגה לבעלת העסק |
| `recommended_questions` | הכנת טיוטת שאלות |
| `status` | סימון מצב הפנייה |
| `needs_human_review` | עצירה לבדיקה אנושית |

## תרגול קצר

כתוב JSON פשוט לפנייה של לקוח.

השתמש בשדות הבאים:

- `customer_name`
- `email`
- `service_type`
- `message`
- `urgent`
- `missing_details`

תבנית:

```json
{
  "customer_name": "",
  "email": "",
  "service_type": "",
  "message": "",
  "urgent": false,
  "missing_details": []
}
```

לאחר מכן ענה:

1. איזה שדה הוא טקסט?
2. איזה שדה הוא boolean?
3. איזה שדה הוא array?
4. איזה שדה היית שולח ל-AI?
5. איזה שדה היית שומר במערכת?

כתוב משפט סיכום:

"המידע על הפנייה יעבור בין המערכות כ-JSON, כך שכל מערכת תדע איפה נמצא [...], איפה נמצא [...], ומה צריך לעשות בהמשך."

## טעויות נפוצות / נקודות זהירות

- לשכוח מרכאות כפולות סביב מפתחות.
- לשים פסיק אחרי השדה האחרון.
- לערבב טקסט ומספרים בלי להבין את ההבדל.
- לשים ערך boolean בתוך מרכאות, למשל `"true"` במקום `true`, כאשר רוצים ערך לוגי.
- לשלוח ל-AI מידע לא מסודר ואז להתפלא שהתוצאה לא עקבית.
- לא להגדיר שמות שדות ברורים.
- להכניס מידע רגיש או API keys לתוך דוגמאות.
- לחשוב שצריך לדעת לתכנת כדי להבין JSON בסיסי.

## חיבור לפרויקט המרכזי

בשיעור הזה נגדיר את מבנה המידע הבסיסי של פנייה בפרויקט המרכזי.

משימה לפרויקט:

1. כתוב JSON לדוגמה של פנייה חדשה.
2. כלול פרטי לקוח בסיסיים, סוג שירות והודעה.
3. כתוב JSON לדוגמה של פלט AI.
4. סמן אילו שדות יישמרו.
5. סמן אילו שדות יוצגו לאדם לבדיקה.
6. ודא שאין בדוגמה מידע רגיש או מפתח API אמיתי.

דוגמה:

- קלט: שם, אימייל, סוג שירות, הודעה.
- פלט AI: סיכום, פרטים חסרים, שאלות המשך, סטטוס.
- בקרה: `needs_human_review` יסמן אם נדרש אישור אדם.
